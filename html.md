一、 前端核心技术栈选型
既然后端是 ThinkPHP 8，为了保证开发效率和后期的易维护性（尤其是未来可能涉及到打包成客户端或 APK 的需求），前端建议采用纯粹的前后端分离架构 (SPA)。

核心框架：Vue 3 (Composition API) + Vite。这是目前国内全栈开发效率最高、生态最契合中小型 SaaS 的组合。

状态管理：Pinia。用于全局管理“当前选中的图片、选中的风格、生成的任务队列”等状态。

UI 组件库：Element Plus 或 Radix Vue。用于快速搭建侧边栏、下拉菜单和弹窗。

CSS 框架：Tailwind CSS。AI 绘图工具的面板通常需要极高密度的自定义排版，Tailwind 能让你极速完成像素级还原。

图像交互核心 (关键)：Fabric.js 或原生 HTML5 Canvas。由于详情图需要“模板+图片”合成，前端必须具备强大的画布操控能力（拖拽、缩放、图层叠加）。

二、 PAIUI 核心交互布局设计 (UI/UX)
参考主流商业 AIGC 工具的成熟设计，PAIUI 的主操作界面应采用“经典三栏式工作流布局 (Three-Pane Layout)”，以最大化屏幕利用率：

1. 左侧栏：资产与历史面板 (Assets & History)
核心功能：用户的“素材库”和“历史记录”。

内容构成：

上传区：支持拖拽批量上传原图，前端直接做第一道 FileReader 压缩校验。

历史记录流：按照时间倒序排列已生成的历史任务，点击缩略图可一键复用当时的 Prompt 和参数。

2. 中间区：核心画布与预览台 (The Canvas)
核心功能：沉浸式的视觉处理中心。

交互逻辑：

原图对比：支持类似 BAA 模式（Before/After 滑块），让用户直观看到抠图和换背景前后的对比。

详情图编排：当切换到“详情图生成”模式时，这里变成一个长图的可视化模板，用户可以将左侧生成的单图直接拖拽进指定的模板“坑位”中。

3. 右侧栏：参数控制台 (Control Panel)
核心功能：向后端发送生成指令的控制枢纽。

内容构成：

场景预设 (Styles)：提供分类的场景卡片（如：极简大理石、夏日水波、赛博朋克等），背后映射着不同的 ComfyUI Prompt。

光影与材质调节：通过滑块控制光照强度（映射到底层 IC-Light 的参数）或边缘融合度。

尺寸与比例：预设 1:1 (淘宝主图)、3:4 (小红书)、16:9 等主流电商比例。

全局执行按钮 (CTA)：带有动态进度状态的“立即生成”按钮。

三、 前后端通信与状态机流转 (State Machine)
前端与后端的 ThinkPHP 交互绝不能是简单的 HTTP 请求，必须设计状态轮询，以应对 AI 生图的高延迟：

发起请求：用户点击生成，前端通过 Axios 提交数据到 PHP 的 /api/task/create。

前端占位：收到 PHP 返回的 task_id 后，中间画板立即生成一个骨架屏 (Skeleton) 或带呼吸灯的“生成中...”占位图。

进度轮询：前端启动一个 setInterval，每隔 2 秒向 PHP /api/task/status?id=xxx 询问进度。

状态渲染：

Status 1 (排队中)：显示“算力分配中，您前面还有 X 个任务”。

Status 2 (生成中)：显示“AI 正在重绘光影...”。

Status 3 (已完成)：获取高清图 URL，关闭轮询，渲染图片并加入左侧历史记录。

Status -1 (失败)：弹窗提示错误信息（如“未识别到产品主体”），退回算力额度。

四、 前端代码如何融入现有的 Docker 体系？
在前一版 docker-compose.yml 中，我们为 paiui-app 使用了 php-nginx 镜像。前端项目编译后的静态文件将完美融合进去：

你在本地通过 npm create vite@latest paiui-web 创建前端工程。

开发完成后，运行 npm run build，会生成一个 dist 文件夹。

把 dist 文件夹内的所有内容，拷贝到你 ThinkPHP 项目的 /public 目录下。

Nginx 会直接将请求指向 /public/index.html，前端路由 (Vue Router) 接管页面，而以 /api 开头的请求则由 Nginx 转发给底层的 index.php 进行业务处理。

这样，你最终交付给客户的依然只有那个清爽的 Docker 容器包，前端被完美“封装”在了里面。
